# Task #5 — Prepare Benchmark Evaluation Sets

## Overview

This script prepares four evaluation sets used to test the trained models. These correspond to the five columns in Table 1 of the paper. None of these sets are used during training — they are test-only.

## How to Run

```bash
pip install datasets
python prepare_eval_sets.py
```

Output will be saved in `./eval_sets/`.

## Pipeline

### Step 1 — GSM-Plus (distractor-insertion subset)

```python
ds = load_dataset("qintongli/GSM-Plus", split="test")
```

GSM-Plus has 8 perturbation types across 10,552 examples. We keep only the `"distraction insertion"` subset (1,319 examples), where a distracting sentence is inserted into each problem.

This is an **OOD** evaluation — the distractors are generated by an LLM, independently from the template-based distractors used in training.

Each record:
```json
{
  "id": 0,
  "question": "...[original question + distractor sentence]",
  "answer": "...\n#### 72",
  "gold_answer": 72.0,
  "perturbation_type": "distraction insertion"
}
```

---

### Step 2 — GSM-DC

```python
ds = load_dataset("YMinglai/GSM-DC-Dataset-Sample", split="train")
```

GSM-DC constructs math problems as directed acyclic graphs (DAGs) and injects distractor nodes at controlled structural positions. We use the sample dataset (6,300 problems) across three noise levels (light / medium / hard).

This is an **OOD** evaluation — the DAG-based distractor construction differs structurally from the template-based training distractors.

Each record:
```json
{
  "id": 0,
  "question": "...[problem text with distractors]",
  "solution": "...[solution steps]",
  "gold_answer": 3.0,
  "irrelevant_nodes_count": 5
}
```

> Note: The `gold_answer` here comes from `problem_info['final_answer']` in the original dataset, not from `####` format like GSM8K.

### Step 4 — Info-Dense GSM8K subset

```python
ds = load_dataset("gsm8k", "main", split="test")
```

Filters the GSM8K test split (1,319 problems) to keep only problems where every sentence that contains a number has that number appear in the solution steps. These are "information-dense" problems where nothing is redundant.

Used to test **over-filtering** — if a model trained on distractors starts ignoring relevant information, accuracy on this subset will drop.

**Filtering logic:**
1. Split the question into sentences
2. Extract numbers from the solution steps (before `####`) — these are the numbers actually used
3. For each question sentence that contains numbers: check if those numbers appear in the solution
4. If any sentence has numbers not used in the solution → discard the problem
5. Also requires at least 4 sentences to ensure meaningful length

Result: 530 / 1,319 examples pass the filter.

Each record:
```json
{
  "id": 5,
  "question": "...[question where every number is necessary]",
  "answer": "...\n#### 48",
  "gold_answer": 48.0
}
```

## How These Are Used

After training the five model instances (Task #11), each model is evaluated on all four test sets. The accuracy results fill in Table 1 of the paper:

```
model accuracy on this test set
robustness score = ─────────────────────────────────
                   model accuracy on clean GSM8K

over-filtering score = distractor-trained accuracy on info-dense
                       ─────────────────────────────────────────
                       clean-trained accuracy on info-dense
```
